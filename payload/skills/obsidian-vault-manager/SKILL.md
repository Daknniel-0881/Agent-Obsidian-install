# Obsidian Vault Manager — 知识库管理 Skill (v3.1)

> 个人知识库的收录、检索、整理、Wiki Link 自动维护的统一入口。
> 架构哲学：**4+1 层信息分层 + PARA + P.A.I.R + 冷藏区隔离 + 命名前缀规范 + 高频场景路由 + Wiki Link 自动识别**。
> v3.1 关键升级（2026-05-09）：**新增 Wiki Link 自动识别 SOP（8 维实体扫描 + 7 步流程），新建/更新文件后强制扫描候选 wikilink，自动建立向上/横向/向下三层链接**。
> v3 升级（2026-05-08）：4+1 层信息分层模型 + L1 个人经验层 + 高频场景路由表 + frontmatter 强制 layer 字段。
> v2 升级：99-冷藏区、命名规范、Grep 默认豁免规则。

## 触发时机

- 用户说"收录到知识库"、"存到Obsidian"、"入库"
- 用户提问时需要检索知识库
- 整理/归类/移动知识库文件
- 补充或维护 Wiki Link、双向链接、加链接、补链接（v3.1 自动触发）
- 创建任何新笔记（v3.1 强制扫描 wikilink）
- 创建新笔记或更新 MOC 索引
- 从冷藏区淘金提炼

## 信息分层模型（4+1 层 · v3 新增）

> 基于余一个人知识库 4 层分类法 + 用户第 5 层冷藏的扩展。详见 [[[方法论] 信息分层模型]] 与 [[2026-05-08-余一-个人知识库4层分类法]]。

| 层 | 定义 | 物理目录 | frontmatter |
|---|-----|---------|-------------|
| **L1 个人经验层** | 用户原话/亲历/一手感受/带个体烙印的判断 | `Areas/个人经验/` | `layer: experience` |
| **L2 在场素材层** | 自己在场的录音/会议/对话原文 | `07-Archive/播客转录/`、`Areas/AI行业/AI播客/` | `layer: scene` |
| **L3 外部入口层** | 收件箱、未分类、未消化原料 | `00-Inbox/` | `layer: inbox` |
| **L4 萃取成品层** | 加工过的成品/半成品 | `Areas/`、`04-Concepts/`、`05-Playbooks/`、`06-Library/`、`07-Archive/已结案项目/` | `layer: extract` |
| **L5 冷藏原料层** | 不删除但不主动检索的低频原料 | `99-冷藏/` | `layer: cold` |

### 分层关键规则

1. **PARA（目录维度）按主题分，分层（frontmatter 维度）按加工度分**——两套结构正交，一篇笔记同时属于"某主题 + 某加工层"
2. **个人经验层（L1）是最稀缺的**——一手原话/感受不能被转发文章淹没，目录物理隔离
3. **L1 ≠ 创业日记**——日记是为了沉淀 SOP（L4），个人经验是保留原汁原味的"我"（L1）
4. **萃取层不存原料**——L4 只存自己/AI 萃取过的成品
5. **冷藏不删除**——L5 默认不进入检索，等待季度淘金

## 核心原则

### 1. 先收再理（P.A.I.R）
- 分类不确定 → 先扔 `00-Inbox/`
- 定期整理 Inbox → 归位

### 2. 原料与成品分离（PARA）
- **成品**（Areas ~ 06-Library）：经过消化提炼的知识
- **高价值原料**（07-Archive 播客/论文）：保留在主区
- **低频原料**（99-冷藏）：默认不检索，偶尔淘金

### 3. 项目有终点，领域没终点
- `01-Projects/` 完成后 → 方法论提炼到 `05-Playbooks/`，记录归档到 `07-Archive/已结案项目/`
- `Areas/` 只做加法，持续积累

### 4. 原文神圣不可侵犯
- 收录原文一字不改
- AI 分析追加在原文之后的独立章节
- 朋友圈素材只读不写，序号不动

### 5. 复利优先
- 每次入库都要思考：这个知识能不能复用？
- 能提炼为 SOP/方法论 → 同步写入 `05-Playbooks/`
- 能抽象为概念 → 同步创建/更新 `04-Concepts/` 节点

### 6. 冷藏不删除（v2 新增）
- 99-冷藏区是隔离不是丢弃
- 冷藏区文件原文一字不改
- 高价值发现 → 复制（不剪切）到主区，原文件加 `cherry_picked: true`

## 命名规范（v2 新增）

### 04-Concepts 节点命名

格式：`[前缀] 主题名.md`（全简体中文 + 必带前缀）

| 前缀 | 含义 | 示例 |
|------|------|------|
| `[MOC]` | 索引页 / 多文件汇总 | `[MOC] 用户分层与定价体系.md` |
| `[理论]` | 概念定义 / 原理 | `[理论] 知识库与RAG完全指南.md` |
| `[工程]` | 具体实现 / 技巧 | `[工程] 稀缺性推理在上下文工程中的应用.md` |
| `[方法论]` | SOP 之上的元方法论 | `[方法论] 知识库管理.md` |

**冗余词黑名单**（不得用于节点名）：调研、报告、深度、完全、详解、全景

**英文节点名**：除专有名词（NVIDIA、API、Harness、Hermes、Karpathy 等）外，全部翻译成中文。专有名词附中文说明。

### Areas 文件命名

- 全中文，不带前缀（前缀仅用于 04-Concepts）
- 日期类用 `YYYY-MM-DD-主题.md`

### Frontmatter 强制字段(v3 升级)

```yaml
---
tags: [主标签, 辅助标签]
created: YYYY-MM-DD
layer: extract  # v3 必填:experience/scene/inbox/extract/cold,详见信息分层模型
source: <URL 或来源说明>
author: <作者>
cherry_picked: false  # 仅冷藏区淘金后才设 true
---
```

**layer 字段速查**:
- `experience`:用户原话/亲历/一手感受(L1,放 `Areas/个人经验/`)
- `scene`:自己在场的录音/会议原文(L2,放 `07-Archive/播客转录/` 或 `Areas/AI行业/AI播客/`)
- `inbox`:收件箱、未分类(L3,放 `00-Inbox/`)
- `extract`:加工过的成品/半成品(L4,默认值,主区大部分文件用此)
- `cold`:冷藏低频原料(L5,放 `99-冷藏/`)

## 收录流程

### Step 1: 判断内容类型 → 选择目录

```
内容是什么？
├── 具体项目交付物 → 01-Projects/
├── 某领域的深度内容 → Areas/对应子目录/
├── 关于某个人的深度拆解 → 03-People/
├── 可跨域复用的核心概念 → 04-Concepts/（必带 [前缀]）
├── 可复用的方法论/SOP → 05-Playbooks/
├── 工具/资源/失败案例/论文/网络/课程 → 06-Library/对应子目录/
├── 已结案项目 / 高价值原料（播客/论文）→ 07-Archive/对应子目录/
├── 推特/公众号/朋友圈/活动文案 → 99-冷藏/对应子目录/
└── 不确定 → 00-Inbox/
```

### Step 2: 创建笔记

```markdown
---
tags: [主标签, 辅助标签]
created: YYYY-MM-DD
source: <来源URL或说明>
author: <作者>
---

# 标题

> 一句话摘要

## 正文内容

（原文完整保留）

## AI 分析与梳理

（AI 的补充分析，与原文明确分隔）

## 相关链接

- [[相关概念1]]
- [[相关文档2]]
```

### Step 3: 建立 Wiki Link

每篇精炼笔记必须：
1. **向上链接**：链接到所属领域的 MOC（`_MOC-*.md`）
2. **横向链接**：链接到相关概念节点（`04-Concepts/`）
3. **向下链接**：链接到引用的具体文档

### Step 4: 更新 MOC 索引

- 新笔记加入对应目录的 `_MOC-*.md`
- 新增子领域 → 创建新 MOC
- `!_MOC/知识库总览.md` 必要时同步更新

## 检索流程（v2 重要更新）

### 默认 Grep / Glob 命令模板

```bash
# 主区检索（默认豁免冷藏 + 系统目录）
grep -r "QUERY" $VAULT_ROOT \
  --exclude-dir=99-冷藏 \
  --exclude-dir=_system \
  --exclude-dir=.obsidian \
  --exclude-dir=.omc \
  --exclude-dir=.smart-env \
  --exclude-dir=Assets \
  --exclude-dir=Excalidraw

# 淘金检索（显式声明，仅冷藏）
grep -r "QUERY" $VAULT_ROOT/99-冷藏

# 全量检索（极少用，需明确目的）
grep -r "QUERY" $VAULT_ROOT \
  --exclude-dir=.obsidian --exclude-dir=.omc --exclude-dir=.smart-env
```

### 检索三步法

1. **目录路由**：根据问题主题匹配目录（参考路由表）
2. **关键词 Grep**：在匹配目录内搜，默认豁免冷藏
3. **Wiki Link 跳转**：读取 `[[]]` 链接追溯关联知识

### 检索豁免

纯代码执行、文件操作、≤20 字简单确认不需要检索。

### 检索结果引用

融入回答时标注：「根据知识库 [[XX文档]]...」

## 淘金流程（v2 新增）

### 触发时机

- AI 判断某主题在主区无果，但冷藏区可能有
- 用户明确说"翻一下推特/公众号有没有 XX"

### 淘金 SOP（3 步）

1. **定位**：在 `99-冷藏/` 内显式 grep 关键词
2. **评估**：核对目标文件 frontmatter 的 source/author/date，判断是否值得提炼
3. **迁移**：
   - **复制**（不是 mv）到 Areas 对应子目录
   - 原冷藏文件 frontmatter 加 `cherry_picked: true`
   - 主区版本加 Wiki Link 反向指向 `99-冷藏/原文件`
   - 主区版本添加「AI 分析」章节

### 淘金禁忌

- 不要批量从冷藏区往主区搬（违背冷藏初衷）
- 同一主题已有主区文件 → 在已有文件追加引用，不新建

## Wiki Link 自动识别与维护（v3.1 升级 · 强制）

### 触发时机（必须自动执行）

每次以下操作完成后**必须**自动触发 wikilink 扫描：

- 创建任何新文件（frontmatter 写完后，正文落定后）
- Edit 已有文件且追加内容 ≥ 50 字
- 用户明确说"加 wikilink"、"补双向链接"、"加链接"
- 回写 AI 分析到知识库时

### 自动识别 SOP（8 维实体扫描）

新建/更新文件后，必须按以下 8 维度主动扫描候选 wikilink：

| 维度 | 实体类型 | 候选目录 | 链接形态 |

### 7 步自动扫描流程

```
Step 1: NER 候选词提取
  - 从正文识别专有名词、概念名、工具名、人物名
  - 从 frontmatter tags 提取主题词
  - 从段落标题、列表项首词提取

Step 2: vault 内 Glob/Grep 候选词
  ls $VAULT_ROOT/04-Concepts/ | grep -iE "<候选词>"
  ls $VAULT_ROOT/03-People/*/ | grep -iE "<候选词>"
  ls -R $VAULT_ROOT/05-Playbooks/ | grep -iE "<候选词>"
  ls -R $VAULT_ROOT/Areas/ | grep -iE "<候选词>"

Step 3: 命中分级
  - 强匹配（全词匹配文件名）→ 必加 wikilink
  - 中匹配（文件名包含候选词）→ 推荐加 wikilink
  - 弱匹配（仅 tags 重合 ≥ 2）→ 助手判断
  - 不匹配 → 不创建死链

Step 4: 在「## 相关链接」追加 [[]]
  - 没有「## 相关链接」区块 → 末尾创建
  - 已有 → 按维度分组追加，去重

Step 5: 反向链接补充（推荐）
  - 在被链接文件追加反向引用（若该文件存在「## 相关链接」区块）
  - 不写入冷藏区文件

Step 6: MOC 同步
  - 文件落到的目录有 _MOC-*.md → 在 MOC 加索引行
  - 新建的概念节点 → 同步注册到「!_MOC/知识库总入口.md」

Step 7: 链接质量门槛验证
  - L4 萃取层 ≥ 2 个 wikilink
  - 04-Concepts 节点 ≥ 5 个，至少 3 跨域
  - L1 个人经验层 ≥ 1 个（向上指 _MOC-个人经验）
  - 06-Library 工具清单 ≥ 1 个
  不达标 → 二次扫描或报告给用户决定
```

### 链接形态规范

| 强度 | 形式 | 用途 |
|------|------|------|
| 强链接 | `[[文件名]]` | 内容核心相关，直接引用 |
| 别名链接 | `[[文件名\|别名]]` | 文件名长/不通顺时用别名 |
| 反向链接 | 双向写入 | 跨域知识网枢纽 |

### 链接质量标准（v2 提高 · v3.1 巩固）

| 文件类型 | wikilink 数量要求 |
|---------|------------------|
| `04-Concepts/` 节点 | **≥ 5，至少 3 个跨域**（跨 Areas/03-People/05-Playbooks 等） |
| `Areas/` 文件（L4） | ≥ 2 |
| `Areas/个人经验/` L1 文件 | ≥ 1（向上指 `_MOC-个人经验`） |
| `06-Library/` 工具清单 | ≥ 1（指向相关 SOP 或概念） |
| `07-Archive/` 高价值原料 | 建议 ≥ 1 |
| `99-冷藏/` | 不强制 |


### Wikilink 禁止事项

1. **不创建死链**：写 `[[XX]]` 前必须 Glob/Grep 确认目标存在
2. **不重复链接**：同一文件同一目标只链一次
3. **不污染冷藏**：`99-冷藏/` 内不强制 wikilink，反向引用也不写入冷藏
4. **不创建无意义弱链接**：tags 重合 < 2 个 → 不强加链接
5. **不在 frontmatter 内写 wikilink**：仅在正文「## 相关链接」或正文段落

### 批量补充方法（已存量文件）

1. 读取目标文件
2. 识别概念、人物、工具、项目（走 8 维扫描）
3. 检查目标实体是否已有节点
4. 在「相关链接」区块追加 `[[]]`
5. 实体不存在且重要 → 创建新概念节点（带前缀）

## 论文入库 SOP（v1.0 · 2026-05-09 新增）

### 触发条件
以下任一情况自动触发论文入库流程：
- 用户发来英文论文 PDF 或 arXiv 链接
- 用户说"帮我入库这篇论文"或"读这篇 paper"
- 用户转发学术推文并附带论文链接

### 强制要求（5 条，缺一不算完成）

1. **双版本**：`paper-EN.md`（英文原文 Markdown）+ `paper-ZH.md`（中文翻译）
2. **中文翻译**：标题、正文、图注、表格、公式说明全部翻译为简体中文；LaTeX 公式原样保留；References 不译（原样保留英文）
3. **图片**：提取到 `images/` 子目录；嵌入语法 `![[images/figure-N.png]]`；嵌入位置与原论文完全一致
4. **统一路径**：`06-Library/精选论文/<论文短名>/`（论文短名用英文或中英混合，见命名示例）
5. **README.md**：入库笔记，结构见下方模板

### 标准目录结构

```
06-Library/精选论文/<论文短名>/
├── README.md          # AI 入库笔记（中文）
├── paper-EN.md        # 英文原文（完整，含 frontmatter + Obsidian 图片嵌入）
├── paper-ZH.md        # 中文翻译（结构与 EN 版完全一致）
├── _原始-paper.pdf    # 原始 PDF（如已下载）
└── images/
    ├── figure-1.png
    ├── figure-2.png
    └── ...
```

### README.md 模板结构

```markdown
## 一句话定位
## 核心贡献（3-5 条）
## 关键数字（参数量、加速比、benchmark 提升等）
## AI 点评（对业务/教学的钩子）
## 文件清单
## 相关链接（arXiv / GitHub / 推文）
```

### frontmatter 强制字段（paper-EN.md 和 paper-ZH.md）

```yaml
---
title: "论文标题"
authors: "作者列表"
venue: "期刊/会议/arXiv"
arxiv_id: "XXXX.XXXXX"
code: "https://github.com/..."
date_published: "YYYY-MM-DD"
date_added: "YYYY-MM-DD"
tags: [关键词1, 关键词2]
layer: extract
---
```

### 论文短名命名规范

- 格式：`<机构或作者>-<核心概念>-<补充词>`
- 示例：`Sakana-Sparser-Faster-Lighter-Transformer`、`LeWorldModel-JEPA非Transformer路线`
- 长度：不超过 50 个字符；英文用连字符分隔；中文关键词可混用

### 完成后必做

1. 更新 `06-Library/精选论文/README.md` 的论文目录表
2. 新增对应行：标题 / 作者 / 关键词 / 入库日期 / wiki link
